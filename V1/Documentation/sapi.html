Sapi

Référence complète du langage — génération de serveurs Axum/Tokio en Rust

Introduction

Sapi (Slyders API) est un langage haut-niveau qui compile des fichiers .sapi en projets Rust complets utilisant Axum (routeur HTTP) et Tokio (async runtime). Un seul fichier source suffit pour définir des bases de données, des services, des routes HTTP et de la logique métier.

Sapi génère du code Rust production-ready : connexions SQL, sérialisation JSON, CORS, logging, Swagger UI, rate limiting et timeouts.

CLI

Le nom de fichier est toujours optionnel — si omis, sapi cherche automatiquement main.sapi dans le dossier courant.

CommandeDescription

sapi build --linuxCross-compile un binaire Linux musl x86_64 (statique) depuis Windows.

Les dossiers de travail locaux sont regroupés sous .sapi/ à la racine du projet : .sapi/build/tmp, .sapi/run/out et .sapi/watch/out.

Configuration runtime (sapi.conf)

Le runtime généré lit automatiquement un fichier sapi.conf (ou le chemin défini par la variable d'environnement SAPI_CONF).

log.level=error
  log.to_stderr=true
  log.to_file=true
  log.error_path=logs/sapi-error.log
  log.critical_path=logs/sapi-critical.log
  log.rotate.max_bytes=10485760
  log.rotate.max_files=3
  panic.include_backtrace=false

log.level accepte critical, error, warn, info, debug (seuil minimum à conserver).

Déploiement sur serveur (nginx)

# Sur la machine de développement — produit dist/ avec le binaire et le log
sapi build
# → dist/sapi_server  (binaire standalone, aucune dépendance)
# → dist/build.log    (journal de compilation)

# Vers un dossier personnalisé
sapi build api.sapi ./prod

# Cross-compiler un binaire Linux depuis Windows (pour déploiement sur VPS/nginx)
sapi build --linux
# → dist/sapi_server  (ELF statique musl, aucune dépendance sur le serveur)

# Copier le binaire sur le serveur puis lancer
./sapi_server

Le binaire compilé est standalone : ni Sapi ni Rust ne sont nécessaires sur le serveur de production. Nginx fait un proxy_pass vers le port défini dans le service.

Prérequis pour --linux (une seule fois)

# Installer Zig (compilateur C/C++ multiplateforme utilisé par cargo-zigbuild)
winget install zig.zig

# Installer cargo-zigbuild (wrapper Cargo → Zig pour cross-compilation)
cargo install cargo-zigbuild

# Ajouter la cible musl x86_64
rustup target add x86_64-unknown-linux-musl

Si l'un de ces prérequis manque, sapi build --linux affiche un message d'erreur clair avec la commande d'installation à lancer.

use — Imports

use "std/http"

Déclare les bibliothèques standard utilisées. "std/http" est la bibliothèque HTTP de base (inclut Axum, Tokio, serde).

database

Déclare une connexion à une base de données. Plusieurs bases peuvent coexister dans un même fichier.

database MainDB {
    type: "postgres"
    host: env("DB_HOST")   // variable d'environnement
    port: env("DB_PORT")
    user: env("DB_USER")
    pass: env("DB_PASS")
    name: "ma_base"
    pool: 10               // taille du pool de connexions
    log {
      path: "logs/sql-main.log"
      level: "info"
      slow: 250           // ms: seuil slow query
    }
}

database LogDB {
    type: "postgres"
    host: "localhost"    // valeur littérale
    port: "5433"
    user: "logger"
    pass: "secret"
    name: "logs"
    pool: 5
}
ChampValeursDescription
type"postgres"SGBD cible (seul PostgreSQL pour l'instant)
hostlittéral ou env("VAR")Hôte de la base
portentier ou env("VAR")Port TCP
userlittéral ou env("VAR")Utilisateur SQL
passlittéral ou env("VAR")Mot de passe
namelittéral ou env("VAR")Nom de la base
poolentierConnexions simultanées max
log { path, level, slow }bloc optionnelActive les logs SQL (requête, durée, slow query). slow est en millisecondes.

service

Un service est un serveur HTTP. On peut en déclarer plusieurs (ports différents, configs différentes).

service UserApi {
    port:    3000
    swagger: true          // active /swagger-ui
    cors:    true          // CORS ouvert (toutes origines)
    logging: "info"
        log {
          path:  "logs/userapi-access.log"
          level: "info"
        }
}

service AdminApi {
    port:    3001
    swagger: false
    cors:    ["https://admin.example.com", "https://backoffice.example.com"]
    limit:   50            // requêtes concurrentes max
    timeout: 30            // secondes avant timeout
    logging: "debug"
}
ChampValeurs possiblesDescription
portentierPort d'écoute TCP (obligatoire)
swaggertrue / falseActive /swagger-ui avec l'UI Swagger
corstrue / "origin" / ["o1","o2"]CORS : all / une origine / liste
limitentierLimite de concurrence (rate limiting)
timeoutentier (secondes)Timeout de requête
logging"info" "debug" "warn" "error"Niveau de log tracing
log { path, level }bloc optionnelActive les logs d'accès HTTP (route, IP, user-agent, status, durée). Sans ce bloc, aucun access log n'est émis.

Le bloc log utilise la même politique de rotation que sapi.conf (log.rotate.max_bytes et log.rotate.max_files).

Routes HTTP

Une route est associée à un service par son nom. La syntaxe est :

// NomDuService  méthode  "chemin"  { corps }

/// Doc de la route (apparaît dans la doc générée)
UserApi get "/users" {
    // corps de la route
}

UserApi post   "/users"        { /* ... */ }
UserApi put    "/users/{id}"    { /* ... */ }
UserApi patch  "/users/{id}"    { /* ... */ }
UserApi delete "/users/{id}"    { /* ... */ }

Les paramètres de chemin :param sont automatiquement convertis en {param} dans l'OpenAPI généré.

Info — Chaque route peut lire ses variables locales, appeler des fonctions, exécuter du SQL et retourner une valeur.

Body & validation

La syntaxe body actuelle est clean-only avec des blocs field. Les formes legacy en ligne ne sont plus recommandées.

Body JSON

Api post "/users" {
  body json {
    field string name {
      required.onError("name requis")
      min(2).onError("name trop court")
    }
    field string email {
      required.onError("email requis")
      email.onError("email invalide")
    }
    field int? age {
      min(18).onError("age min 18")
    }
    field object address {
      field string city { required.onError("ville requise") }
      field string zip {}
    }
  }

  return 201: { string status = "created" }
}

Body Form

Api post "/contact" {
  body form {
    field string name {}
    field string message {}
  }
  return "ok"
}

Body Form Multipart

Api post "/upload" {
  body form multipart {
    field string title {}
    field string fileName {}
  }
  return "uploaded"
}
Règles — Les objets imbriqués sont supportés en body json. Ils ne sont pas supportés en body form ni en body form multipart.

Headers & cookies

Dans une route, tu peux lire des headers et cookies HTTP avec des déclarations explicites. Les deux formes existent en obligatoire et optionnelle.

Syntaxe

// Header obligatoire (400 si absent ou invalide)
header string authHeader = "Authorization"

// Header optionnel (string vide si absent)
header string? lang = "Accept-Language"

// Cookie obligatoire (400 si absent)
cookie string sessionId = "session_id"

// Cookie optionnel (string vide si absent)
cookie string? theme = "theme"

Exemple complet

MonApi get "/profile" {
  header string auth = "Authorization"
  cookie string? sessionId = "session_id"

  return {
    string token = auth,
    string sid = sessionId
  }
}
Comportement — Les déclarations string (sans ?) retournent une erreur 400 structurée si la valeur est absente. Les déclarations string? produisent une chaîne vide si la donnée n'existe pas.

Auth JWT & guard

Sapi permet de déclarer un provider JWT via auth, de signer des tokens avec Auth.sign(...) et de protéger des routes avec guard.

Déclaration d'un provider auth

auth Auth {
    secret:    env("JWT_SECRET")
    algorithm: "HS256"
    expiry:    3600
    claims:    { int userId, string role, string email }
}
ChampDescription
secretClé de signature JWT (souvent via env("JWT_SECRET"))
algorithmAlgorithme de signature (ex. "HS256")
expiryDurée de validité du token en secondes
claimsSchéma typé des claims accessibles dans les routes protégées

Guard hiérarchique (roles + priorité dynamique)

auth Auth {
  secret:    env("JWT_SECRET")
  algorithm: "HS256"
  expiry:    3600
  claims:    { int userId, string role, string email }
  guard {
    key: "rank"
    guard_op: ">="
    roles: [
      { role: "member", rank: 10 },
      { role: "moderator", rank: 50 },
      { role: "admin", rank: 100 },
      { role: "owner", rank: 999, super: true }
    ]
  }
}
Champ guardDescription
keyNom de la clé numérique de priorité dans chaque rôle (rank, weight, priority, ...)
guard_op / opOpérateur de comparaison entre rôle utilisateur et rôle requis : >=, >, ==, <=, <
rolesListe des rôles connus avec leur valeur de priorité dynamique
super: trueBypass total des checks de rôle (ex. owner/super-admin)

Signer un token

AuthApi post "/login" {
  body json {
    field string username {}
    field string role {}
  }
  string token = Auth.sign({
    int userId = 1,
    string role = role,
    string email = username
  })
  return { string token = token }
}

Protéger une route avec guard

// JWT valide requis
AuthApi get "/me" {
  guard Auth
  return {
    int id = userId,
    string role = role,
    string email = email
  }
}

// JWT valide + rôle requis
AuthApi get "/admin" {
  guard Auth "admin"
  return "Zone administrateur"
}

Forme courte possible si le provider auth est unique (ou non ambigu) :

AuthApi get "/mod" {
  guard "moderator"
  return "Zone modérateur"
}
Info — Dans une route avec guard Auth, les claims déclarés dans auth Auth { claims: ... } sont injectés comme variables locales (ex. userId, role, email).
Comportement — Sans token ou avec token invalide, la route protégée renvoie 401. Avec un rôle requis, un token valide mais insuffisant est refusé en 403. Si auth.guard est défini, la comparaison suit guard_op et la priorité portée par key.
Codes d'erreur — Les routes auth/guard utilisent les codes runtime unifiés auth_missing_header, auth_invalid_token et guard_forbidden.

Fonctions

Les fonctions sont déclarées au niveau racine du fichier et peuvent être appelées depuis n'importe quelle route.

Fonction synchrone

fn greet(string name) {
    string upper = name:upper
    string msg   = "Bonjour, {upper}!"
    return msg
}

// Appel dans une route
UserApi get "/hello" {
    string result = greet("Alice")
    return result
}

Fonction asynchrone

async fn fetchLabel(string key, string lang) {
    string upper = key:upper
    string label = "{upper} [{lang}]"
    return label
}

// Appel avec await dans une route
UserApi get "/label" {
    await string result = fetchLabel("welcome", "fr")
    return result
}
Note — Les paramètres de fonction n'acceptent que des types scalaires (string, int, float, bool, date, time). Pas de list:T ni de json en paramètre.

Déclarations de variables

string nom    = "Alice"
int    age    = 28
float  note   = 18.5
bool   actif  = true

Le compilateur émet let mut par défaut pour permettre la réassignation.

lock — Immutabilité

Le mot-clé lock marque une variable comme immuable (let sans mut en Rust). Obligatoire sur les listes et JSON que vous ne voulez pas modifier.

lock string     version  = "1.0.5"
lock list:string tags     = ["rust", "axum", "sapi"]
lock json        config   = { "env": "prod", "debug": false }
Erreur — Appeler :push() sur une liste lock est une erreur de compilation.

Chaînes & Interpolation

Les chaînes sont délimitées par des guillemets doubles. Les variables sont interpolées avec {nom}.

string prenom = "Alice"
string msg    = "Bonjour, {prenom} !"   // → "Bonjour, Alice !"
string multi  = "{prenom} a {age} ans"  // plusieurs variables
Note — Seuls les types scalaires (string, int, float, bool, date) peuvent être interpolés. Les variables json et list ne peuvent pas apparaître dans {…}.

Séquences d'échappement

string s1 = "ligne 1\nligne 2"   // \n : saut de ligne
string s2 = "tab\there"          // \t : tabulation
string s3 = "guillemet : \""     // \" : guillemet

Listes

Une liste est typée : le type de l'élément suit list:.

list:string fruits = ["pomme", "poire", "cerise"]
list:int    scores = [10, 20, 30]
list:string vide   = []                 // liste vide (mutable)

lock list:float prix = [9.99, 14.99]   // immuable

Accès par index

string premier = fruits[0]   // "pomme"
int    troisieme = scores[2]  // 30

Méthodes de liste

MéthodeRetourDescription
:countintNombre d'éléments
:firstT?Premier élément (optionnel)
:lastT?Dernier élément (optionnel)
:push(val)—Ajoute un élément (liste mutable)
int    total   = fruits:count       // 3
string dernier = fruits:last        // "cerise"

output:push("banane")

JSON

Les objets JSON sont des littéraux clé/valeur. Les valeurs doivent être des littéraux ou des chaînes interpolées.

string env = "production"
int    ver = 2

lock json info = {
    "env":     "{env}",   // variable string via interpolation
    "version": 2,           // littéral entier
    "ratio":   99.9,        // littéral float
    "active":  true         // littéral bool
}
Règle importante — Dans un objet JSON, les valeurs ne peuvent PAS être des idents nus ("env": env est interdit). Pour insérer une variable, utilisez l'interpolation de chaîne : "env": "{env}".

date & time

date today = new date()   // date courante (YYYY-MM-DD)
int  nowMs  = new time()   // timestamp Unix en millisecondes

string msg = "Aujourd'hui : {today}"
string ts  = "Timestamp : {nowMs}"

Méthodes chaînées

La syntaxe variable:methode applique une transformation. Les méthodes peuvent être enchaînées et acceptent des arguments.

Règle — La chaîne doit commencer par un identifiant de variable. On ne peut pas chaîner directement sur un littéral : "hello":upper est interdit, mais msg:upper est valide.

Enchaînement multiple

string raw     = "  hello world  "
string cleaned = raw:trim:upper              // "HELLO WORLD"

bool  found   = raw:trim:contains("hello") // true

string slug   = raw:trim:lower:replace(" ", "-") // "hello-world"

Dot-access JSON

Navigue dans un objet json avec la notation point. Le résultat est toujours une string. Des méthodes chaînées peuvent être appliquées ensuite.

lock json user = {
    "name":    "Bob Dupont",
    "email":   "bob@example.com",
    "country": "Canada"
}

string name    = user.name                   // "Bob Dupont"
string email   = user.email                  // "bob@example.com"
string upper   = user.name:upper             // "BOB DUPONT"
bool   isCA    = user.country:starts_with("Can") // true

Accès par index

lock list:string items = ["a", "b", "c"]

string premier  = items[0]   // "a"
string deuxieme = items[1]   // "b"

// Avec une variable comme index
int    idx      = 2
string element  = items[idx]   // "c"

Appels de fonction

string msg    = greet("Alice")
int    longueur = nameLength(nom)
string slug   = buildSlug("Mon Titre")

// Plusieurs arguments
await string label = fetchLabel("welcome", "fr")

Les noms de fonctions sont automatiquement convertis en snake_case dans le Rust généré (nameLength → name_length).

Réassignation

Une variable peut être réassignée (si elle n'est pas lock) avec la syntaxe nom = nouvelleValeur.

string name = "world"
string msg  = greet(name)   // "Bonjour, world!"

name = "sapi"               // réassignation
msg  = greet(name)         // "Bonjour, sapi!"

if / else / else if

Les branches conditionnelles permettent d'exécuter du code différent selon une condition. On peut chaîner autant de else if que nécessaire. Le else final est optionnel.

Syntaxe

if condition {
    // branche vraie
} else if autre_condition {
    // branche intermédiaire
} else {
    // branche par défaut
}

Opérateurs de comparaison

OpérateurSignification
==Égal à
!=Différent de
<Inférieur strict
>Supérieur strict
<=Inférieur ou égal
>=Supérieur ou égal

Exemple — classification numérique

Api get "/classify/{n}" {
    route { int n }
    if n > 0 {
        return "positive"
    } else if n < 0 {
        return "negative"
    } else {
        return "zero"
    }
}

Exemple — retour avec code HTTP

Api get "/check/{code}" {
    route { int code }
    if code == 200 {
        return 200: { string result = "ok" }
    } else if code == 404 {
        return 404: { string result = "not found" }
    } else {
        return 400: { string result = "bad code" }
    }
}

Exemple — if sans else (réassignation)

Api get "/even/{n}" {
    route { int n }
    string msg = "odd"
    if n == 2 {
        msg = "even"   // réassignation dans le bloc if
    }
    return msg
}

Exemple — comparaison de chaîne

Api get "/greet/{lang}" {
    route { string lang }
    if lang == "fr" {
        return "Bonjour"
    } else if lang == "es" {
        return "Hola"
    } else {
        return "Hello"
    }
}
Note — Un bloc if peut contenir n'importe quelle instruction : déclarations, réassignations, appels SQL, retours. Le compilateur génère du Rust idiomatique avec des blocs conditionnels.

Boucle for

Itère sur une liste. La syntaxe utilise :for directement sur la variable liste.

Sans index

lock list:string fruits = ["pomme", "poire", "cerise"]

fruits:for string fruit {
    string upper = fruit:upper
    output:push(upper)
}

Avec index

fruits:for string fruit, int i {
    string ligne = "{i}: {fruit}"
    output:push(ligne)
}

Exemple complet

UserApi get "/tags" {
    lock list:string base      = ["rust", "axum", "sapi"]
    list:string      formatted = []

    base:for string tag, int i {
        string entry = "{i}: {tag}"
        formatted:push(entry:upper)
    }

    int    count = formatted:count
    string first = formatted[0]

    return {
        int    count = count,
        string first = first
    }
}

push

Ajoute un élément à une liste mutable (non-lock).

list:string resultat = []

resultat:push("premier")
resultat:push("deuxième")
// resultat = ["premier", "deuxième"]

return

Deux formes de retour : valeur unique ou multi-champs (objet JSON en réponse).

Valeur unique

return msg          // retourne une variable
return "ok"         // retourne un littéral
return rows         // retourne un résultat SQL (Vec de struct)

Multi-champs (JSON structuré)

Déclare chaque champ de la réponse avec son type et sa valeur. Génère un struct Rust sérialisé en JSON.

return {
    int    total   = count,
    string status  = msg,
    bool   healthy = alive,
    float  ratio   = pct
}

Résultat HTTP : 200 application/json avec le corps {"total": 42, "status": "ok", ...}

await

Attend le résultat d'une fonction asynchrone ou d'une requête SQL.

Fonction async

await string label = fetchLabel("welcome", "fr")
// type   nom     = appel()

Requête SQL

await User rows   = getUsers().many()       // Vec<User> — sans arguments
await User user   = getFirst().one()        // User (erreur si absent)
await Run  result = insertUser({ name: name, email: email }).run() // avec arguments
// StructType  nom = sqlVar().méthode()             — {} optionnel si aucun placeholder
// StructType  nom = sqlVar({ clé: val }).méthode()  — avec liaisons nommées

Requêtes SQL — Déclaration

Une requête SQL est d'abord déclarée avec sql, puis exécutée avec await … .one() | .many() | .run().

SELECT avec colonnes de sortie

sql getUsers = `SELECT id, name, email FROM users ORDER BY name` : {
    int:id, string:name, string:email
} use MainDB

SELECT avec placeholder

sql searchUsers = `SELECT id, name FROM users WHERE email LIKE {domain}` : {
    int:id, string:name
} use MainDB

await SearchResult results = searchUsers({ domain: domain }).many()

INSERT / UPDATE / DELETE (sans sortie)

sql insertUser =
    `INSERT INTO users (name, email) VALUES ({name}, {email})`

await InsertResult r = insertUser({
    name:  name,
    email: email
}).run()

DELETE sans arguments

sql purge = `DELETE FROM users WHERE active = false`
await PurgeResult r = purge().run()    // {} omis — aucun placeholder
Syntaxe de la déclarationExplication
sql nom = `QUERY`Query SQL en backticks
: { type:col, … }Colonnes de sortie (obligatoire pour SELECT)
use NomDBBase de données cible (optionnel, défaut = première déclarée)

Requêtes SQL — Exécution

MéthodeRust généréUsage
.one()fetch_one()Attend exactement 1 ligne (erreur si absent)
.many()fetch_all()Retourne toutes les lignes (Vec<T>)
.run()execute()Requête d'écriture, retourne le nb de lignes affectées

Arguments nommés

Les placeholders {nom} dans la query SQL sont liés par nom dans l'appel :

sql updateUser =
    `UPDATE users SET name = {newName}, email = {newEmail} WHERE id = {userId}`

await UpdateResult r = updateUser({
    newName:  newName,
    newEmail: newEmail,
    userId:   userId
}).run()
Sécurité — Sapi génère des requêtes paramétrées ($1, $2, …) via SQLx. L'injection SQL est impossible grâce aux bindings typés.

Accolades optionnelles

Lorsqu'une requête SQL ne contient aucun placeholder {…}, les accolades peuvent être omises dans l'appel :

sql getAll  = `SELECT id, name FROM users` : { int:id, string:name }
sql purge   = `DELETE FROM logs WHERE old = true`

await User rows = getAll().many()   // ✓ {} omis
await Res  r    = purge().run()    // ✓ {} omis

// Les deux formes sont équivalentes :
await User rows2 = getAll({}).many()  // ✓ {} explicite (rétrocompatible)

Schéma généré automatiquement

Pour chaque sql avec colonnes de sortie, Sapi génère un struct Rust :

// Pour : sql getUsers = `…` : { int:id, string:name, string:email }
// Sapi génère :
#[derive(sqlx::FromRow, serde::Serialize)]
pub struct GetUsers {
    pub id:    i64,
    pub name:  String,
    pub email: String,
}

Checklist de configuration (prod)

Cette section sert de mémo rapide pour éviter les oublis avant mise en production d'une API Sapi.

1) Blocs obligatoires selon le besoin

BlocObligatoireChamps minimumNotes
serviceOui (toujours)portswagger recommandé à false en prod publique.
databaseSi SQL utilisétype, host, port, user, pass, namePrivilégier env("...") pour les secrets.
authSi routes protégéessecret, algorithm, expiry, claimsguard { ... } active les règles hiérarchiques de rôles.
sqlSi accès DBrequête + schéma retour + use DBNombre de placeholders {...} = nombre d'arguments passés à l'appel.

2) Service & runtime

  • service.port défini et libre.
  • service.timeout défini pour éviter les requêtes pendantes.
  • service.logging cohérent avec l'environnement (info ou warn en prod).
  • Si besoin d'access logs HTTP: bloc service.log { path, level }.
  • sapi.conf présent (ou variable SAPI_CONF) avec rotation de logs configurée.

3) Auth, headers, cookies

  • Routes sensibles protégées avec guard.
  • header string ... pour valeurs obligatoires (ex: Authorization).
  • header string? ... ou cookie string? ... pour valeurs optionnelles.
  • Codes d'erreur attendus vérifiés: 401 et 403.

4) Body validation

  • Toutes les entrées externes passent par body json ou body form.
  • Règles explicites sur les champs critiques (required, min, etc.).
  • Objets imbriqués uniquement en body json (pas en body form).

5) SQL & robustesse

  • Chaque requête SQL testée au moins en mode .one(), .many() ou .run() selon le cas.
  • Blocs database.log activés pour diagnostiquer les lenteurs (slow).
  • Gestion des erreurs runtime couverte côté client API (sql_error, validation_error, etc.).
Conseil release — Avant publication, exécuter la suite _TEST_/run_all_tests.ps1, vérifier bugs.md, puis regénérer la doc depuis LANGUAGE.html vers la version Distribution active.

Documentation — sapi doc

La commande sapi doc génère un fichier OpenAPI 3.1 à partir de votre fichier .sapi. Avec --html, elle produit également un site interactif Scalar.

Commandes

# JSON OpenAPI sur stdout
sapi doc mon_api.sapi

# Site HTML complet (works offline, double-clic)
sapi doc mon_api.sapi --html --output docs

Doc-comments

Les commentaires /// juste au-dessus d'une route, d'un service ou d'une fonction sont extraits comme summary dans l'OpenAPI :

/// Retourne la liste complète des utilisateurs actifs
/// Les résultats sont triés alphabétiquement par nom.
UserApi get "/users" {
    // ...
}

La première ligne devient le summary, les lignes suivantes la description.

Inférence de schéma

Les schémas de réponse sont déduits automatiquement :

Type de retourSchéma OpenAPI

return rows (SQL .many){"type": "array", "items": {...}}
return user (SQL .one){"$ref": "#/components/schemas/NomStruct"}
return { int x = … }{"type": "object", "properties": {...}}

Exemple complet minimal

use "std/http"

database DB {
    type: "postgres"
    host: env("DB_HOST")
    port: env("DB_PORT")
    user: env("DB_USER")
    pass: env("DB_PASS")
    name: "mydb"
    pool: 5
}

service Api {
    port:    3000
    swagger: true
    cors:    true
    logging: "info"
}

fn slugify(string text) {
    string lower = text:lower
    string slug  = lower:replace(" ", "-")
    return slug
}

/// Retourne le statut de l'API
Api get "/health" {
    string status = "ok"
    date   today  = new date()
    return {
        string status = status,
        string date   = "{today}"
    }
}

/// Liste tous les articles
Api get "/articles" {
    sql getAll = `SELECT id, title, content FROM articles ORDER BY id` : {
        int:id, string:title, string:content
    } use DB

    await Article rows = getAll().many()
    return rows
}

/// Crée un article
Api post "/articles" {
    string title   = "Mon article"
    string slug    = slugify(title)
    string content = "Contenu de l'article."

    sql insert =
        `INSERT INTO articles (title, slug, content) VALUES ({title}, {slug}, {content})`

    await InsertResult r = insert({
        title:   title,
        slug:    slug,
        content: content
    }).run()

    return "Article '{title}' créé"
}
